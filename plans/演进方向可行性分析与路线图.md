# PRD Tools 演进方向可行性分析与路线图

> 基于全网企业级 AI 工程化优秀案例调研
> 分析时间：2026-04-29
> 工具版本：v2.4.0

## 调研来源

| 来源 | 关键发现 |
|------|---------|
| Anthropic Agent Patterns | 3 种 Workflow + 4 种 Agentic Pattern（Orchestrator-Workers、Evaluator-Optimizer、Prompt Chaining、Routing） |
| Gartner 2025-2026 | 40% 企业应用将内嵌 AI Agent（2025 约 5%，增长 ~8x） |
| Deloitte 模块化智能体 | Agent-powered 模块化工作流成为企业标准 |

| CrewAI Multi-Agent | 不同角色 Agent 分别处理各自领域，多轮协商消除不一致 |

## 方向 1：多轮构建，优化前后端各自 reference

**可行性：高 | 投入：中 | 收益：高**

### 行业映射

- Anthropic 的 **Evaluator-Optimizer Pattern**：Agent 生成初稿 → 评估器打分 → 根据差距反馈迭代优化，直到质量达标
- CrewAI 的 **Multi-Agent Refinement**：不同角色的 Agent 分别处理各自领域，通过多轮协商消除不一致
- 当前 `build-reference` 是单轮生成，reference 质量高度依赖首次 prompt 质量

### 具体方案

```text
Round 1: 全量构建（现有流程）
    ↓
Round 2: 各层自检 — 每层独立检查 boundary 违规、证据缺失、
         前后端契约引用是否对齐
    ↓
Round 3: 跨层对齐 — 前端 reference 的 client_contract 与
         后端 reference 的 api_surface 交叉验证
    ↓
Round 4: 质量门控（现有 step-03）
```

### 关键收益

- 前端/BFF/后端各自独立维护 reference，不再被迫写入一个共享但边界模糊的知识库
- 跨层契约通过 `contract_ref` 引用而非重复，符合 SSOT
- 多轮迭代天然适配 Evaluator-Optimizer 模式，有成熟的 prompt engineering 经验

### 风险点

- 多轮意味着 token 成本上升（约 2-3x）
- 需要设计"何时停止迭代"的收敛条件（建议：boundary 违规数 < 阈值 或 达到最大轮次）

## 方向 2：沉淀 B 端营销知识库

**可行性：中高 | 投入：高 | 收益：长期战略价值**

### 行业映射

- Gartner：到 2026 年底，40% 企业应用将内嵌 AI Agent
- Salesforce Agentforce：行业级知识库 + Agent 自动化，垂直领域知识是核心壁垒
- HubSpot Breeze AI：营销场景的 AI Copilot，依赖结构化营销知识（活动类型、权益模型、奖励机制）

### 具体方案

```text
阶段 1（当前已有基础）：
  05-domain.yaml 的 terms + implicit_rules + decision_log
  → 已经是领域知识的雏形

阶段 2：抽取为独立知识库
  marketing-knowledge-base/
  ├── activities.yaml      # 活动类型、生命周期、状态机
  ├── benefits.yaml        # 权益模型、发放规则、互斥策略
  ├── rewards.yaml         # 奖励类型、预算模型、审计规则
  ├── coupons.yaml         # 券模板、核销、过期策略
  ├── approvals.yaml       # 审批流、风控规则
  └── integrations.yaml    # 外部系统契约（支付、消息、风控）

阶段 3：跨项目共享
  多个项目（dive-bff、genos、ibt-genos-benefit）
  共享同一份营销知识库，各自的 reference 通过 ref 引用
```

### 关键收益

- 跨项目 PRD 分析不再从零开始，营销域知识直接复用
- 新人 onboarding 成本大幅下降
- 与方向 3 形成正向飞轮：知识库越丰富 → 能力面适配器越精准 → 输出质量越高

### 风险点

- 初期投入大，需要从多个项目的 reference 中提炼共性
- 知识库维护成本——需要建立更新机制（每次 prd-distill 后的回流是天然入口）
- 不确定性：如果团队业务线变化快，知识库可能很快过时

## 方向 3：prd-tools 工具迭代，优化能力面适配器

**可行性：最高 | 投入：低 | 收益：中高**

### 行业映射

- Anthropic 的 **Routing Pattern**：根据输入特征动态路由到不同处理流程
- 当前能力面适配器（`layer-adapters.md`）是静态的文本描述，所有项目共用一套
- Deloitte：模块化 Agent 工作流的核心是"每个模块有明确的输入/输出契约"

### 具体方案

```text
当前状态：
  references/layer-adapters.md（通用适配器，前端/BFF/后端各一套固定能力面）

优化方向：
  1. 动态适配器 — 根据 project-profile.yaml 的技术栈自动选择能力面
     例：React 项目启用 ui_route + form_or_schema
         Vue 项目启用不同的组件模式

  2. 项目级自定义 — 允许在 _reference/project-profile.yaml 中
     声明自定义能力面（如 data-pipeline、ab-test）

  3. 适配器版本化 — 随 prd-tools 版本演进，旧项目不受影响

  4. 从方向 2 的营销知识库反哺 — 当知识库沉淀了足够多的
     "活动类型→能力面命中模式" 后，自动更新路由信号
```

### 关键收益

- 低风险、高回报的渐进式优化
- 适配器质量直接影响 prd-distill 输出精度
- 与方向 1、2 天然衔接

### 风险点

- 单独做方向 3 的边际收益有限，需要结合方向 1（多轮构建）才能真正验证适配器质量

## 方向 4：Skill 调用产出三方视角的 B 端整体 report + plan

**可行性：高 | 投入：中 | 收益：最高（领导最关注的方向）**

### 行业映射

- Anthropic 的 **Orchestrator-Workers Pattern**：中央编排器拆分任务 → 分发给专业子 Agent → 汇总结果
- 这是方向 4 的精确映射：一个 orchestrator skill 调用 frontend/backend/BFF 的 prd-distill，汇总为三方视角的统一 report

### 架构设计

```text
/prd-distill-full（新 skill）
    │
    ├── Orchestrator（主控）
    │   ├── 读取 PRD + 技术方案
    │   ├── 拆分为各层子任务
    │   └── 汇总最终输出
    │
    ├── Worker: Frontend Agent
    │   ├── 读取 genos/ 的 _reference/
    │   ├── 生成前端层 layer-impact + contract-delta
    │   └── 返回结构化结果
    │
    ├── Worker: BFF Agent
    │   ├── 读取 dive-bff/ 的 _reference/
    │   ├── 生成 BFF 层结果
    │   └── 返回结构化结果
    │
    └── Worker: Backend Agent
        ├── 读取后端服务的 _reference/
        ├── 生成后端层结果
        └── 返回结构化结果

最终输出：
  report.md  — 三方视角：前端改什么、BFF 改什么、后端改什么、
               跨层契约有哪些风险、哪些 TODO 未映射
  plan.md    — 前端/BFF/后端各自的详细技术方案，
               精确到文件路径+行号+验证命令
  questions.md — 跨团队需对齐的阻塞项
```

### 输出增强（领导提到的具体要求）

| 输出项 | 对应内容 |
|--------|---------|
| PRD 理解 | 从 requirement-ir 汇总的三方视角需求解读 |
| 能力点映射 | 各层命中的能力面清单，标注已映射/未映射 |
| 未映射的 TODO | PRD 提到但当前代码库没有对应能力面的需求点 |
| Test Case | 从 QA 矩阵汇总的三方测试用例 |
| Question | 跨团队契约冲突、owner 未确认项 |
| 技术方案详情 | 各层精确到文件路径的开发计划 |

### 关键收益

- 一次 skill 调用产出全栈视角，不需要分别跑 3 次 prd-distill 再人工对齐
- Orchestrator-Workers 是 Anthropic 官方推荐的多 Agent 模式，有成熟实践
- 对领导的价值最大：一个 PRD 直接产出完整的三方技术方案

### 风险点

- 需要方向 1 先落地（各层有独立 reference），否则 orchestrator 没有高质量的子 Agent 输入
- 多 Agent 编排的 token 成本（估算：单次 ~3-5x 单层 prd-distill）
- 需要处理各层结果之间的冲突（如前端假设的接口与后端实际接口不一致）

## 推荐演进路线图

```text
Phase 1（1-2 周）：方向 3 — 能力面适配器优化
  ├─ 低风险快速见效
  ├─ 动态适配器 + 项目级自定义能力面
  └─ 为后续方向打基础

Phase 2（2-4 周）：方向 1 — 多轮构建
  ├─ build-reference 支持 2-3 轮迭代
  ├─ 各层独立 reference + 跨层契约引用
  └─ Evaluator-Optimizer 模式实现

Phase 3（4-8 周）：方向 4 — 三方视角 Skill
  ├─ 依赖 Phase 2 的独立 reference
  ├─ Orchestrator-Workers 架构
  ├─ prd-distill-full skill 开发
  └─ 三方 report + plan 输出

Phase 4（持续）：方向 2 — 营销知识库
  ├─ 从 Phase 2 的多项目 reference 中提炼
  ├─ 随每次 prd-distill 持续回流
  └─ 长期战略资产
```

### 依赖关系

```text
方向 3（适配器优化）
    ↓ 不阻塞但增强
方向 1（多轮构建）────→ 方向 4（三方视角 Skill）
                              ↑ 依赖
方向 2（营销知识库）←──── 持续沉淀 ←──── 全程
```

Phase 1-3 有明确的依赖关系和递进逻辑：**适配器精准 → reference 质量高 → 三方编排效果好**。方向 2 是贯穿全程的持续沉淀，不阻塞其他方向。
