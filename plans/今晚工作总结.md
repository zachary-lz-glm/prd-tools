## 事件 1：Reference 知识库去重优化（v2.3）

### 问题

老板反馈 reference 生成的知识边界模糊、内容重叠严重。经分析：

- 旧版 10 个文件混用 3 种分类维度（内容性质 / 关注点 / 使用场景），同一条知识平均出现 2-5 次
- 根因：模板字段语义重叠 + prompt 无去重检查 + 7 个模板缺少 boundary 声明
- 全网调研证实：行业共识是 **SSOT（Single Source of Truth）**，知识去重是 RAG / AI 知识库的最佳实践，重叠会降低检索质量和生成准确性

### 做了什么

**架构重设计：10 文件 → 6 文件**

```text
旧结构（v3.1, 10 文件）          新结构（v4.0, 6 文件）
├── 00-index.md                  ├── 00-portal.md（人类导航 + 场景阅读指南）
├── 01-entities.yaml             ├── 01-codebase.yaml（静态清单：目录、枚举、模块）
├── 02-architecture.yaml    ─┘   ├── 02-coding-rules.yaml（编码规则：规范+约束合并）
├── 03-conventions.yaml     ─┼─→ ├── 03-contracts.yaml（契约：字段级唯一权威来源）
├── 04-constraints.yaml     ─┘   ├── 04-routing-playbooks.yaml（路由信号 + 场景打法）
├── 05-routing.yaml         ─┐   └── 05-domain.yaml（业务领域：术语+背景+决策日志）
├── 06-glossary.yaml        ─┼─→
├── 07-business-context.yaml ─┘
├── 08-contracts.yaml       ──→
└── 09-playbooks.yaml       ─┘
```

**三个核心机制：**

1. **Boundary 声明**：每个文件模板顶部声明"只放什么、不放什么"，从源头消除分类歧义
2. **跨文件 ID 引用**：字段级信息只存在于 `03-contracts.yaml`，其他文件通过 `contract_ref` 引用，不再重复
3. **去重检查**：step-02 末尾增加 5 条显式去重规则，生成完成后自动扫描违规项

**变更文件：**
- 重写 6 个模板文件（`templates/`）
- 新建 `references/reference-v4.md`（6 文件边界定义 + 归档优先级规则 + v3.1 兼容映射）
- 重写 `steps/step-02-deep-analysis.md`（6 阶段生成顺序 + 去重检查步骤）
- 重写 `steps/step-03-quality-gate.md`（5 条 boundary 检查项）
- 更新 `SKILL.md`、`workflow.md` 等引用文件
- 删除旧版 10 个模板 + `reference-v3.md`

### 预期收益

| 指标 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| 同一知识重复次数 | 2-5 次 | 1 次 | 80%+ |
| 总信息量 | ~88KB / 2200 行 | ~55KB / 1500 行 | -37% |
| AI context window 浪费 | ~20% 重复内容 | 低 | -20% token |
| 维护同步点 | 平均 2-5 处 | 1 处 | 60%+ |

### 行业参考

- **Diátaxis Framework**：文档按"学习目标"分类（Tutorial / How-to / Reference / Explanation），而非按内容载体——这启发了我们按"知识在开发生命周期中的角色"单一维度分类
- **SSOT（Single Source of Truth）**：数据库设计、微服务 API 契约管理中的成熟实践，适用于 AI 知识库
- **Boundary-driven Architecture**：借鉴 DDD 中 Bounded Context 的理念，每个 reference 文件是一个明确边界的知识上下文

## 事件 2：渐进式披露输出优化（v2.4）

### 问题

对比 v2.2 和 v1.0 的 prd-distill 输出质量：

- v2.2 的"大局观"更好（架构分层、契约分析、证据链），但 report 只有 68 行，细节被压到 YAML artifacts 里
- v1.0 的可读性更好（详细的 TODO list、字段清单、操作步骤），但缺乏架构层面的分析
- 根因：v2.2 过度追求"人机分离"，把人需要的信息也放进机器可读的 YAML 了

### 做了什么

**引入 Progressive Disclosure（渐进式披露）模式**

参考 Nielsen Norman Group 的信息架构方法论，在同一个 `report.md` 文件内实现从结论到细节的逐层展开：

```text
report.md（渐进式披露结构）
├── 1. 需求摘要（30 秒决策）    ← 一句话 + 变更类型统计
├── 2. 影响范围                  ← 命中的层、能力面、关键文件
├── 3. 关键结论                  ← 带 REQ-ID 和代码路径
├── 4. 变更明细表                ← 所有 IMP-* 项，精确到文件路径
├── 5. 字段清单                  ← 按模块分组，含类型/必填/契约 ID
├── 6. 校验规则                  ← 规则 + 错误文案 + 目标文件
├── 7. 开发 Checklist            ← 可直接勾选的操作列表
├── 8. 契约风险                  ← needs_confirmation / blocked 项
└── 9. Top Open Questions        ← 最多 5 个阻塞问题
```

**plan.md 增强：**

- 精确到文件路径 + 行号
- `- [ ]` checklist 格式，开发可直接勾选
- 每个任务包含：目标文件、操作描述、参考实现、关联 REQ/IMP/CONTRACT、验证命令
- 按 Phase 分组，Phase 间标注依赖

**变更文件：**
- 重写 `plugins/prd-distill/references/output-contracts.md`（完整的 report.md + plan.md 模板定义）
- 重写 `plugins/prd-distill/steps/step-03-confirm.md`（9 章节渐进式披露规格 + plan 可执行手册规格）
- 更新 `plugins/prd-distill/workflow.md`、`SKILL.md`

### 核心设计决策

- **不是把 YAML 展开，而是重新设计信息层次**：每一层都是独立的阅读终点，不需要跳到其他文件
- **保留 artifacts 证据链**：report.md 是人类入口，artifacts/ 仍然是机器可读的审计和回流基础
- **结合 v1.0 的具体性 + v2.2 的架构性**：每个变更项都带文件路径和代码引用，但组织方式是架构驱动的

### 行业参考

- **Progressive Disclosure**（Nielsen Norman Group）：复杂系统界面设计中的经典方法论——先给决策摘要，再逐层展开细节
- **Diátaxis 的 How-to Guide**：面向任务、可执行的步骤化文档，对应 plan.md 的 checklist 格式
- **Google Design Doc 文化**：技术方案文档应该一屏可读结论，按需展开细节——对应 report.md 的"30 秒决策"章节

## 事件 3：演进方向可行性分析与路线图

### 背景

领导提出 4 个后续演进方向，要求结合全网企业级 AI 工程化优秀案例做可行性分析。

### 分析结论

| 方向 | 可行性 | 投入 | 收益 | 核心行业映射 |
|------|--------|------|------|-------------|
| 1. 多轮构建优化前后端 reference | 高 | 中 | 高 | Anthropic Evaluator-Optimizer Pattern |
| 2. 沉淀 B 端营销知识库 | 中高 | 高 | 长期战略 | Salesforce Agentforce 行业知识库 |
| 3. 优化能力面适配器 | 最高 | 低 | 中高 | Anthropic Routing Pattern |
| 4. 三方视角 B 端整体 report + plan | 高 | 中 | 最高 | Anthropic Orchestrator-Workers Pattern |

### 推荐路线图

```text
Phase 1（1-2 周）：方向 3 — 能力面适配器优化
  ├─ 低风险快速见效
  ├─ 动态适配器 + 项目级自定义能力面
  └─ 为后续方向打基础

Phase 2（2-4 周）：方向 1 — 多轮构建
  ├─ build-reference 支持 2-3 轮迭代
  ├─ 各层独立 reference + 跨层契约引用
  └─ Evaluator-Optimizer 模式实现

Phase 3（4-8 周）：方向 4 — 三方视角 Skill
  ├─ 依赖 Phase 2 的独立 reference
  ├─ Orchestrator-Workers 架构
  ├─ prd-distill-full skill 开发
  └─ 三方 report + plan 输出

Phase 4（持续）：方向 2 — 营销知识库
  ├─ 从 Phase 2 的多项目 reference 中提炼
  ├─ 随每次 prd-distill 持续回流
  └─ 长期战略资产
```

### 关键行业数据

- Gartner：2026 年底 40% 企业应用将内嵌 AI Agent（2025 年约 5%，增长 ~8x）
- Anthropic 官方推荐的 4 种 Agentic Pattern 中，有 3 种直接映射到我们的演进方向
- Salesforce Agentforce 的核心壁垒不是模型能力，而是垂直行业知识库的深度——这验证了方向 2 的长期价值
