# ADR-0004：口径一致性修复（P0）

| 字段 | 值 |
|------|---|
| 状态 | 已实施 |
| 版本 | v2.4.1 |
| 日期 | 2026-04-29 |
| 提交 | 534939f |
| 触发 | GPT-5.5 review 指出版本口径未完全收敛 |

## Context（为什么做）

GPT-5.5 对 v2.3/v2.4 的 review 结论：

> 这轮优化方向是对的。但现在还不能算完全收敛，最大问题是 v2.4 的架构变了，部分旧文档/旧 prompt 还停留在 v2.2/v3.1，会影响真实使用稳定性。口径不一致等于运行逻辑不一致。

发现的 4 个 P0 问题：

| 问题 | 位置 | 影响 |
|------|------|------|
| 默认 prompt 还写构建 v3.1 reference | `openai.yaml` | 用户/模型可能继续生成旧结构 |
| 上下文收集仍映射到旧文件 | `step-00-context-enrichment.md` | golden sample 会流向 05/08/09 旧体系 |
| 输出契约标题和样例仍是 v2.2 | `output-contracts.md`（两处） | v2.4 发布可信度下降 |
| 阅读指南还描述 10 文件旧 reference | `OUTPUT_READING_GUIDE.md` | 新用户会被文档带偏 |

## Decision（做了什么）

### 1. 版本号统一（6 个文件）

| 文件 | 修复 |
|------|------|
| `openai.yaml` | `v3.1 reference` → `v4.0 reference` |
| `step-00-context-enrichment.md` | 整段重写：v3.1 → v4.0，旧文件映射更新 |
| `step-01-structure-scan.md` | `schema_version: "3.1"` → `"4.0"` |
| `reference/output-contracts.md` | 标题 v2.2 → v2.4，所有 schema/tool 版本更新 |
| `prd-distill/output-contracts.md` | 标题 v2.2 → v2.4，所有 schema/tool 版本更新 |
| `templates/project-profile.yaml` | `schema_version: "3.1"` → `"4.0"` |

### 2. 文档内容更新（1 个文件）

`OUTPUT_READING_GUIDE.md`：reference 章节从 10 文件旧结构更新为 6 文件 v4.0，增加按场景阅读表。

### 3. 输出边界约束（1 个文件）

`step-03-confirm.md`：增加 report/plan/questions 的职责边界和长度约束：

| 文件 | 定位 | 长度约束 |
|------|------|---------|
| `report.md` | 决策文档，不是所有细节全集 | 200-400 行 |
| `plan.md` | 执行手册，不是二次报告 | 150-350 行 |
| `questions.md` | 阻塞清单，不能变成垃圾桶 | 30-80 行 |

## Consequences（影响）

### 收益

- 所有 prompt、模板、文档统一到 v2.4/v4.0，消除运行逻辑不一致
- 输出文件有了明确的长度和职责边界，防止从"太抽象"反弹成"太长太满"

### 仍需关注（GPT-5.5 提出的 P1 建议）

- 增加 reference 重复事实检测脚本或检查清单（只靠 prompt 不够稳定）
- 用同一个 PRD 跑 v1.0/v2.2/v2.4 回归对比
- 建立 golden sample 评分表
