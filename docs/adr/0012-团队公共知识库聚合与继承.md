# ADR-0012：团队公共知识库聚合与继承

| 字段 | 值 |
|------|---|
| 状态 | 计划中 |
| 日期 | 2026-05-13 |
| 起始版本 | v2.18.1（脚手架） |
| 目标版本 | v2.19（聚合） / v2.20（团队 distill） |
| 范围 | PRD Tools 从单仓自治 → 全团队共享知识库 |
| 触发 | 跨仓 PRD（前端+BFF+后端）在单仓 distill 时无法产出跨层技术方案，需要各仓 reference 聚合为团队公共知识库 |

## Context

### 问题

当前 `/reference` 和 `/prd-distill` 都是单仓自治——每个仓库独立构建自己的知识库、独立蒸馏 PRD。但实际 PRD 落地通常涉及 3 个仓（前端 genos、BFF dive-bff、后端 magellan）：

- 单仓 distill 只能看到本仓的 reference，无法感知其他仓的契约和实现
- 跨仓接口契约（如 BFF 的 API 字段 vs 前端消费）缺乏统一视图
- 团队级术语（如 CampaignType 枚举值）在各仓独立定义，容易漂移

### v2.18.1 脚手架已落地

以下字段已在 v2.18.1 的 schema 中定义，为聚合铺路：

- `team_reference_candidate: true` — 标记适合升级到团队仓的条目
- `consumer_repos` / `producer_repo` / `checked_by[]` — 跨仓契约三元组
- `handoff_surfaces` / `cross_repo_handoffs` — 跨层 playbook 交接面
- `team_scope.related_repos` — 团队级关联仓

## Decision

### 决策 1：Pull-based 聚合

团队仓按需从成员仓拉取 reference 聚合，不做 push（每次 push 都要处理冲突、认证复杂、误操作风险高）。

### 决策 2：单业务一仓，按层组织

每个业务域一个独立的团队仓（如 `dive-team-reference`），仓内按层组织（frontend/bff/backend/team）。不再嵌套业务子目录——仓名已表达业务归属。

```
dive-team-reference/
├── team/                    # 聚合产物（SSOT）
│   ├── 01-codebase.yaml     # 跨仓代码坐标索引（不聚合源码细节）
│   ├── 02-coding-rules.yaml # 只聚合 severity=fatal 规则
│   ├── 03-contracts.yaml    # 跨仓契约合并（producer/consumers/checked_by）
│   ├── 04-routing-playbooks.yaml # 只聚合跨层 playbook
│   ├── 05-domain.yaml       # 术语合并（去重 + divergence 标记）
│   └── project-profile.yaml # layer: team-common
├── {frontend,bff,backend}/snapshots/  # 成员仓镜像（追溯用）
└── build/
    ├── aggregation-report.yaml
    └── conflicts.yaml        # 冲突清单（待人工仲裁）
```

### 决策 3：复用 `/reference` 扩展，不新建 skill

在 `/reference` 新增两个模式：

| 模式 | 何时 | 做什么 |
|------|------|--------|
| T 团队聚合 | 在团队仓执行 | 从成员仓聚合事实到 `team/` |
| T2 团队继承 | 在成员仓执行 | 从团队仓继承公共事实到本仓 |

### 决策 4：团队 distill 复用 `/prd-distill`，自动识别

`/prd-distill` 读 `project-profile.yaml` 的 `layer: team-common` 自动切 team 模式。跳过源码扫描，改为从团队仓 `team/01-codebase.yaml` 的跨仓代码地图查。产出 1 份 team-plan.md + N 份 sub-plan-{repo}.md。

### 聚合策略摘要

| 产物 | 策略 |
|------|------|
| 03-contracts | union_by_id：同 ID 合并 producer/consumers/checked_by，冲突标 conflict |
| 05-domain | union_dedupe：去重合并，定义不同标 divergence |
| 02-coding-rules | fatal_only：只聚合 severity=fatal 规则 |
| 04-routing-playbooks | cross_layer_only：只聚合涉及 2+ 层的 playbook |
| 01-codebase | 不聚合源码，只建 cross_repo_entities 索引 |

## Consequences

### 收益

- 跨仓 PRD 可在团队仓直接跑 `/prd-distill`，产出全栈技术方案
- 团队级术语和契约有唯一真相源（team/ 目录）
- 成员仓通过继承获得跨仓公共知识，单仓 distill 质量提升
- 保留单仓自治：团队仓只沉淀"多仓已 checked_by 的事实"

### 风险与限制

- **聚合依赖成员仓 reference 质量**：首轮跑会暴露"reference 不够厚"的问题
- **冲突需人工仲裁**：Producer 不一致、字段定义漂移等需各仓 owner 确认
- **无实时同步**：pull-based 意味着团队仓数据可能滞后
- **v2.19 只做聚合，不做团队 distill**：完整跨仓蒸馏要到 v2.20
