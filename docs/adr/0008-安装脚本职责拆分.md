# ADR-0008：install.sh 职责拆分（install / doctor / 运行时自检）

| 字段 | 值 |
|------|---|
| 状态 | 已过时（v2.13.0 起移除全部第三方依赖，不再需要 uv/MarkItDown/Graphify/GitNexus） |
| 版本 | v2.12.0 实施，v2.13.0 精简 |
| 日期 | 2026-05-07 |
| 触发 | 用户在 dive-bff 安装时遭遇 GitHub 不可达 + SOCKS 代理对 npm/uv 不生效，暴露当前一键安装的诚实性问题 |

## Context（为什么做）

当前 `install.sh`（v2.11.0，526 行）尝试在一个脚本里完成 4 件事：

1. 装 prd-tools 自己的 skills 和本地辅助脚本（仓库唯一真正"拥有"的东西）
2. 装外部依赖：uv / MarkItDown / Graphify
3. 配置 GitNexus MCP（不真装包，只写 `.mcp.json`）
4. 引导用户配 ANTHROPIC_AUTH_TOKEN

实际暴露的问题：

| 问题 | 位置 | 后果 |
|------|------|------|
| GitNexus 是"声明好的"，不是"装好的" | Step 3、Step 6 | 脚本退出时 `✅ All tools installed`，但首次 Claude Code 启动 `npx -y gitnexus@latest mcp` 仍可能拉不到包，用户无感知 |
| MarkItDown / Graphify 失败仅 WARNING | Step 2、Step 4 | 退出码仍是 0，自动化无法判断真实状态 |
| 代理覆盖不全 | Step 1 探测段（install.sh:31-77） | 导出的 `http_proxy/https_proxy` 即便是 `socks5://` 也对 npm（不读 `socks5_proxy`）和 uv（reqwest 默认无 socks 特性）不生效；curl 能下 tarball ≠ uv/npm 能拉包 |
| GitHub tarball 是硬失败点 | Step 5（install.sh:198） | 网络挂时前面装好的 uv/markitdown/graphify 全部白装，无法续跑 |
| 交互式 read API Key 与"安装"职责耦合 | Step 7（install.sh:382-425） | 安装阶段写 shell profile，且只在有 tty 时才生效 |

根因：**这一个脚本同时承担了"必装"、"按需装"、"诊断"、"配置"四种语义不同的职责**，因此每一种失败模式都被混在同一条流水线里，最后 final summary 里 ✅ 和 ❌ 同屏，用户和自动化都无法可信地判断"装好了没"。

## Decision（做了什么）

把 install.sh 拆成三层，每层职责单一、退出码可信。

### 1. install.sh 只做"prd-tools 自己拥有的事"

**保留**：
- 拉 GitHub tarball（`v2.0.tar.gz`） → 复制 reference / prd-distill 到 `.claude/skills/`
- 复制 `doctor.sh` / `status.sh` 到目标项目 `.prd-tools/`
- 写 `.prd-tools-version` 标记文件
- 探测代理用于 curl 拉 tarball（**保留**，但仅声明"用于下载 prd-tools 本身"，不再误导用户以为 uv/npm 也走代理）

**移除**：
- Step 1 装 uv
- Step 2 装 MarkItDown
- Step 3 探测 GitNexus runtime
- Step 4 装 Graphify
- Step 6 写 `.mcp.json`
- Step 7 交互式 API Key 提示和 shell profile 写入
- final summary 里的 `✅ All tools installed`

**结果**：
- 脚本 ≤ 100 行
- 退出码 0 严格意味着"prd-tools skills 和本地辅助脚本已就位"
- 失败只有一种：GitHub 不可达 → exit 1 + 给出 git clone fallback

### 2. 新增 `prd-tools doctor`（独立可执行脚本）

放在 `scripts/doctor.sh`，并由 `/reference` 引导用户在缺依赖时运行。**只诊断，不安装**，输出形如：

```
uv         ✅ /Users/didi/.local/bin/uv (0.4.18)
markitdown ❌ not installed
            → uv tool install "markitdown[all]" && uv tool install markitdown-ocr
graphify   ❌ not installed
            → uv tool install graphifyy && graphify install
gitnexus   ⚠️  declared in .mcp.json, never verified
            → npx -y gitnexus@latest --version    (verifies npm reachability)
ANTHROPIC_AUTH_TOKEN  ❌ unset
            → export ANTHROPIC_AUTH_TOKEN=sk-ant-xxx
proxy      ⚠️  socks5://127.0.0.1:8119
            note: npm/uv 不走 socks，PyPI/npm 拉包可能失败；建议另开 HTTP 代理或镜像
```

支持 `--strict` 模式：任何 `❌` 升级为 exit 1（CI/自动化用）。
支持 `--fix` 模式：可选交互式按表执行修复命令（每条独立确认）。

### 3. skill 运行时自检（按需触发）

`/reference` 和 `/prd-distill` 在真正需要某个工具的步骤前就近自检：

| 触发场景 | 检查项 | 缺失时行为 |
|---------|-------|----------|
| 读 PDF / DOCX / PPTX | `markitdown` 在 PATH | 停在该步，提示 `prd-tools doctor` |
| 跑 `--mode deep` 业务语义图 | `graphify` 在 PATH + `ANTHROPIC_AUTH_TOKEN` | 停，给 fix 命令 |
| 调用代码图谱 | MCP `list_repos` 返回非空 | 停，提示 `npx gitnexus@latest analyze .` |
| 解析 PRD 图片 | `ANTHROPIC_AUTH_TOKEN` 或 `OPENAI_API_KEY` | 降级为纯文本，给提示 |

skill 比安装脚本更清楚"什么时候真的需要哪个工具"，把检查放到使用点，避免一开始就要求全部装齐。

## Consequences（影响）

### 收益

| 维度 | 改善 |
|------|------|
| install.sh 退出码可信 | 成功就是"skills 已装"，不再混入外部依赖状态 |
| 失败定位精确 | doctor 把每一项独立报告，每条 fix 命令可复制 |
| 网络分级 | prd-tools 自身只依赖 GitHub；PyPI/npm 失败不再阻断 install |
| 维护成本下降 | gitnexus 启动参数、graphify 包名变更只改 doctor 一行 |
| 用户认知校准 | 不再有"`✅ All tools installed` 但 gitnexus MCP 加载失败"的认知错位 |

### 风险与权衡

| 风险 | 缓解 |
|------|------|
| "一键体验"打折，部分用户需要多跑一次 `prd-tools doctor` | install.sh 完成时显式提示下一步：`prd-tools doctor` 检查并按表修复 |
| 旧版 install.sh 用户重装时 `.mcp.json` 不再被脚本写入 | doctor 在 GitNexus 项打印 `.mcp.json` 添加片段，用户复制粘贴一次即可 |
| 代理探测从 install 移走后，doctor 也需要复用 | 把代理探测逻辑抽成 `scripts/lib/detect_proxy.sh`，install.sh 和 doctor 都 source |

### 不在本次范围

- 不引入 Python/Node 包管理器作为兜底（保持 shell 单脚本简单性）
- 不做 GUI/Web installer
- 不改 release.sh 和 CHANGELOG 流程

## 落地步骤

| # | 动作 | 产物 |
|---|------|------|
| 1 | 抽取代理探测逻辑 | `scripts/lib/detect_proxy.sh` |
| 2 | 精简 install.sh 到只装 prd-tools 本体 | `install.sh`（≤ 100 行） |
| 3 | 新增 doctor / status 脚本 | `scripts/doctor.sh`、`scripts/status.sh`，安装为项目内 `.prd-tools/doctor.sh`、`.prd-tools/status.sh` |
| 4 | skill 内嵌运行时自检 | 在 reference / prd-distill 各 step 起始处加 dependency-check 段 |
| 5 | 更新 README 安装章节 | 说明三层模型，给出三种典型路径（一键 / 分步 / 离线） |
| 6 | 发布 v2.12.0 | CHANGELOG 引用本 ADR |
