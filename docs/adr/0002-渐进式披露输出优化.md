# ADR-0002：渐进式披露输出优化（report + plan 可读性增强）

| 字段 | 值 |
|------|---|
| 状态 | 已实施 |
| 版本 | v2.4.0 |
| 日期 | 2026-04-29 |
| 提交 | ba461f7 |
| Supersedes | v2.2 输出格式 |

## Context（为什么做）

对比 v2.2 和 v1.0 的 prd-distill 输出质量：

- **v2.2**：大局观更好（架构分层、契约分析、证据链），但 report 只有 68 行，细节被压到 YAML artifacts 里
- **v1.0**：可读性更好（详细的 TODO list、字段清单、操作步骤），但缺乏架构层面的分析
- **根因**：v2.2 过度追求"人机分离"，把人需要的信息也放进机器可读的 YAML 了

## Decision（做了什么）

**引入 Progressive Disclosure（渐进式披露）模式**，在同一个 report.md 文件内实现从结论到细节的逐层展开。

### report.md 9 章结构

```text
├── 1. 需求摘要（30 秒决策）    ← 一句话 + 变更类型统计
├── 2. 影响范围                  ← 命中的层、能力面、关键文件
├── 3. 关键结论                  ← 带 REQ-ID 和代码路径
├── 4. 变更明细表                ← 所有 IMP-* 项，精确到文件路径
├── 5. 字段清单                  ← 按模块分组，含类型/必填/契约 ID
├── 6. 校验规则                  ← 规则 + 错误文案 + 目标文件
├── 7. 开发 Checklist            ← 可直接勾选的操作列表
├── 8. 契约风险                  ← needs_confirmation / blocked 项
└── 9. Top Open Questions        ← 最多 5 个阻塞问题
```

### plan.md 可执行手册

- 精确到文件路径 + 行号
- `- [ ]` checklist 格式
- 每个任务含：目标文件、操作描述、参考实现、关联 REQ/IMP/CONTRACT、验证命令
- 按 Phase 分组，Phase 间标注依赖

### 三文件职责边界

| 文件 | 定位 | 边界 |
|------|------|------|
| `report.md` | 决策文档 | 不展开完整 YAML 证据链，不复制 PRD 原文 |
| `plan.md` | 执行手册 | 不重复 report 结论，不编造行号 |
| `questions.md` | 阻塞清单 | 不放普通备注和已确认结论 |

## Consequences（影响）

### 收益

- 结合了 v1.0 的具体性 + v2.2 的架构性
- 每一层都是独立的阅读终点，不需要跳到其他文件
- 保留了 artifacts 证据链作为审计和回流基础

### 风险

- 可能从"太抽象"反弹成"太长太满"——已增加长度约束（report 200-400 行，plan 150-350 行，questions 30-80 行）

### 变更文件

- 重写 `plugins/prd-distill/references/output-contracts.md`
- 重写 `plugins/prd-distill/steps/step-03-confirm.md`
- 更新 `workflow.md`、`SKILL.md`
