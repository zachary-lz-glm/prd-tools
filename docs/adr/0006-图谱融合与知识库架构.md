# ADR-0006：图谱融合与知识库架构

| 字段 | 值 |
|------|---|
| 状态 | 已实施（图谱融合）/ 已规划（跨项目知识库） |
| 版本 | v2.5.1 |
| 日期 | 2026-05-01 |
| 关联 | ADR-0003（演进路线图）、ADR-0001（SSOT 优化） |

## Context

prd-tools v2.5.0 声称支持 GitNexus + Graphify 图谱增强，但模板、步骤文件和 prd-distill 之间存在三层断裂：

1. 模板只有 `evidence: ["EV-001"]` 占位符，无图谱字段，AI 填模板时不会产出图谱数据。
2. `_output/graph/` 目录从未生成，图谱证据文件不存在。
3. prd-distill 的 step-02-classify 定义了图谱增强字段，但 SKILL.md、workflow.md 和 output-contracts.md 完全没有提到。

同时，多项目（前端 genos、BFF dive-bff、后端 genos-benefit）共享营销领域知识，但每个项目独立构建 `_reference/`，跨项目知识不流通。

## Decision

### 1. 双证据字段设计

每个 reference 条目使用两套独立证据追踪，不能互相替代：

| 字段 | ID 格式 | 用途 |
|------|---------|------|
| `evidence` | EV-xxx | 可审计证据（源码、文档、人工确认） |
| `graph_evidence_refs` | GEV-xxx / GEV-Bxxx | 图谱溯源（GitNexus/Graphify 结构化发现） |

**EV 是审计证据，GEV 是图谱溯源。关键 reference 条目必须至少有 EV 或明确标注豁免原因。**

### 2. graph_sources 数组（非单值）

使用 `graph_sources: []`（数组）而非 `graph_source: ""`（单值），因为：
- `02-coding-rules` 的 rules 可同时来自 GitNexus（结构性规则）和 Graphify（设计原理）
- `04-routing-playbooks` 的 routing 可同时来自 Graphify（聚类模式）和 GitNexus（代码路径验证）
- `danger_zones` 的 blast radius 来自 GitNexus，但 God Nodes 来自 Graphify

### 3. Provider 职责分工

| Provider | 主导 reference 文件 | 辅助 |
|----------|-------------------|------|
| GitNexus | 01-codebase, 03-contracts | 02-coding-rules (rules, danger_zones), 04-routing-playbooks (routing) |
| Graphify | 02-coding-rules (war_stories), 04-routing-playbooks, 05-domain | 02-coding-rules (rules, danger_zones) |

02-coding-rules 是两个 provider 的**交汇处**，沉淀"为什么必须按这个方式写代码"的工程纪律。

### 4. 置信度映射

| 图谱来源 | confidence | 是否需要 EV 源码确认 |
|---------|-----------|-------------------|
| GitNexus AST 提取 | high | 不需要 |
| Graphify EXTRACTED + source locator | high | 不需要 |
| Graphify EXTRACTED 无 locator | medium | 需要 |
| Graphify INFERRED ≥ 0.8 | medium | 需要 |
| Graphify INFERRED < 0.8 | low | 必须确认 |
| Graphify AMBIGUOUS | low | 必须人工确认 |

### 5. graph-sync-report 必须始终产出

`_output/graph/graph-sync-report.yaml` 即使图谱不可用也必须生成，记录 `available: false` 和具体原因（`tool_missing` / `index_missing` / `graph_missing`）。这使质量门控可以区分"没有跑过图谱"和"跑过但不可用"。

### 6. Raw Graph ≠ Reference

**图谱是原始发现层，reference 是精选后的企业知识库。** 图谱原始结果（graph.json、.gitnexus/）不应直接塞入 `_reference/`。必须经过：
- 裁剪（只保留 PRD-to-code 相关的）
- 去重（与 rg/glob 扫描结果合并）
- 确认（Graphify INFERRED 需源码确认）
- 分发（按 reference 文件边界拆分到 01-05）

### 7. prd-distill 轻量消费

prd-distill 只做轻量级补充查询，不做图谱全量扫描：
- GitNexus `impact()` 查受影响符号 → impact 条目的 `affected_symbols`
- Graphify `path`/`explain` 查业务约束 → impact 条目的 `business_constraints`
- 图谱不可用时完全回退到源码 Read + rg/glob

`affected_symbols` 和 `business_constraints` 放在每个 impact 条目内（非全局），保持 REQ/IMP 关联。

## Consequences

### 收益

- 图谱工具的发现能力可直接落地到 reference 模板
- 质量门控可自动校验图谱证据的完整性
- 02-coding-rules 成为 GitNexus + Graphify 的交汇点，沉淀工程纪律
- 为跨项目知识库奠定数据结构基础

### 风险

- 双证据字段增加模板复杂度（graph_sources/graph_evidence_refs 是可选字段缓解此问题）
- 图谱工具不可用时 graph_sources: [] 不构成错误
- 依赖 AI 按文档生成 _output/graph/* 文件，目前没有 deterministic validator

### 跨项目知识库规划（暂不实施）

**设计方向**：

1. 共享 05-domain.yaml：从多项目 reference 中提炼营销领域术语、规则、决策
2. 跨层契约注册表：从各项目 03-contracts 中提取 FE→BFF→BE 契约链
3. Graphify 多项目模式：对多个项目运行 /graphify，合并业务图谱
4. GitNexus group mode：使用 @groupName 查询跨仓库调用链

**前置条件**：

- 单项目图谱融合在至少 2 个项目中验证通过
- 质量门控的图谱校验规则稳定运行
- 团队对 05-domain.yaml 的跨项目维护流程达成共识

**后续需要补的 deterministic validator**：

- graph_evidence_refs 是否都存在
- graph_sources 是否和 graph-sync-report 一致
- Graphify INFERRED 是否误标 high
- 有图谱可用时 reference 是否真的消费了图谱
- 无图谱时是否全部优雅降级为 graph_sources: []
